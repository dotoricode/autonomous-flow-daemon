# 버그 감사 가이드

## 감사 태그 체계

버그 감사가 완료되면 git 태그를 생성하여 다음 감사의 시작점으로 사용합니다.

### 태그 형식
```
BUG-AUDIT-YYYY-MM-DD
```

### 다음 감사 시 절차

1. **변경 파일 식별**:
   ```bash
   git diff --name-only BUG-AUDIT-2026-04-01..HEAD -- src/ tests/
   ```

2. **변경 파일만 감사 대상으로 지정**:
   - 출력된 파일 목록에 대해서만 버그/결함/누수 검사 수행
   - 이전 리포트에서 "미수정" 상태인 항목은 재검증

3. **리포트 작성**:
   - `docs/bug-report/YYYY-MM-DD-<scope>.md` 형식으로 작성
   - 이전 리포트의 ID 체계를 이어서 사용 (C-XX, H-XX, M-XX, L-XX)

4. **태그 갱신**:
   ```bash
   git tag BUG-AUDIT-YYYY-MM-DD
   git push origin BUG-AUDIT-YYYY-MM-DD
   ```

## 심각도 기준

| 등급 | 기준 | 예시 |
|------|------|------|
| CRITICAL | 보안 취약점, 데이터 손실, 크래시 | 경로 탐색, RCE, 미초기화 변수 |
| HIGH | 리소스 누수, 기능 오동작, 중요 누락 | 메모리 누수, fd 미닫힘, 인증 부재 |
| MEDIUM | 성능 저하, 비표준 동작, 엣지케이스 | 캐시 미무효화, TOCTOU, 오파싱 |
| LOW | 코드 품질, 미미한 UX 이슈 | 중복 코드, 미사용 변수, 약한 테스트 |
